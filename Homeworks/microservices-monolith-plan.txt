

1. НАЗНАЧЕНИЕ СИСТЕМЫ И БИЗНЕС-ПРОЦЕССЫ
--------------------------------------------------------------------------------

Назначение: мобильное приложение и веб-сервисы для заказа поездки: пользователь
указывает точки «откуда/куда», система подбирает водителя, строит маршрут,
показывает цену и время, выполняет поездку и списывает оплату.

Бизнес-процесс 1 — «Заказ поездки»:
  1) Пользователь вводит адреса → геокодирование и проверка зоны обслуживания.
  2) Расчёт маршрута и предварительной цены (тарификация).
  3) Поиск свободного водителя, назначение заказа.
  4) Выполнение поездки (трекинг, статусы).
  5) Завершение: фиксация факта поездки, итоговый расчёт, списание с карты/кошелька, чек.

Бизнес-процесс 2 — «Поддержка и безопасность»:
  Проверка пользователя (блокировки, антифрод), обращения в поддержку, инциденты по поездке.


1. МИКРОСЕРВИСЫ 


1) User & Profile
   Учётные записи, профиль, настройки, список платёжных методов (ссылки на PSP).

2) Geo & Routing
   Геокодирование адресов, построение маршрута, ETA, проверка зоны такси.

3) Pricing (Тарификатор)
   Расчёт предварительной и итоговой стоимости по правилам тарифов, скидкам, суржу.

4) Dispatch (Назначение заказов)
   Очередь заказов, подбор водителя по близости/рейтингу, жизненный цикл до «водитель назначен».

5) Payment
   Холд/списание, возвраты, интеграция с платёжным шлюзом.

6) Notification
   Push/SMS о статусах заказа (водитель найден, прибыл, поездка окончена).


2. ВЗАИМОДЕЙСТВИЕ СЕРВИСОВ В ОДНОМ СЦЕНАРИИ


Сценарий: пользователь нажимает «Заказать» после выбора маршрута (до назначения
водителя и резерва оплаты).

1) Клиент → API Gateway / BFF
   Запрос: userId, координаты/адреса A и B, выбранный класс авто.

2) BFF → User & Profile (СИНХРОННО: HTTP или gRPC)
   Передаёт: userId.
   Ожидает: флаг «активен», не в чёрном списке, базовый профиль.
   При блокировке — сценарий обрывается.

3) BFF → Geo & Routing (СИНХРОННО: gRPC)
   Передаёт: точки A, B.
   Ожидает: полилиния маршрута, расстояние (км), ETA (мин), валидность зоны.

4) BFF → Pricing (СИНХРОННО: HTTP REST или gRPC)
   Передаёт: userId, класс тарифа, км, ETA, время суток, город.
   Ожидает: предварительная цена (min–max или фикс), валюта, pricingQuoteId.

5) BFF → Payment (СИНХРОННО: HTTP к платёжному API)
   Передаёт: userId, сумма из quote, pricingQuoteId.
   Ожидает: успех предавторизации (холд) или отказ.

6) BFF → Dispatch (СИНХРОННО: gRPC — создание заказа)
   Передаёт: orderId, координаты, класс, pricingQuoteId, сумма холда.
   Ожидает: orderId, статус «в поиске водителя».

7) Dispatch → Notification (АСИНХРОННО: Kafka или RabbitMQ)
   Событие OrderSearchingDriver: userId, orderId.
   Notification отправляет push «Ищем водителя».

8) Когда водитель принят: Dispatch публикует событие DriverAssigned в шину
   → Notification (push), Trip подписан и создаёт агрегат поездки.

Сводка:
  - Синхронно (gRPC/HTTP): BFF с User, Geo, Pricing, Payment, Dispatch.
  - Асинхронно (Kafka/RabbitMQ): Dispatch → Notification, Trip (события домена).


3. ПАТТЕРН STRANGLER FIG ДЛЯ ОДНОГО МИКРОСЕРВИСА
--------------------------------------------------------------------------------

Кандидат: вынести Pricing (Тарификатор) из монолита.

Идея: внешний контракт не меняется; запросы постепенно перенаправляются со
старого кода на новый сервис.

Шаги:
  1) Фасад в монолите — единая точка IPricingService (логика пока в монолите).
  2) Параллельная реализация — новый микросервис Pricing с тем же контрактом.
  3) Прокси-режим — фасад по feature-flag (1% → 10% → 100%) вызывает либо
     старый модуль, либо HTTP/gRPC к Pricing Service.
  4) Shadow traffic — для части запросов считать цену в монолите и в сервисе;
     логировать расхождения без влияния на пользователя.
  5) Полный перенос — 100% на микросервис, удаление дублирующего кода из монолита.
  6) Монолит остаётся тонким клиентом или BFF проксирует на Pricing.
